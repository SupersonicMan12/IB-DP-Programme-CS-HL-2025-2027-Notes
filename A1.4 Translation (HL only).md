### A1.4.1 Evaluate the transition processes of interpreters and compilers
**The mechanics and use-cases of each translation approach.** **The difference in error detection, translation time, portability and applicability for different translation processes, including just-in-time compilation (JIT) and bytecode interpreters**

**Interpreter:** translates and executes line-by-line
- **must re-translate each run**: **slow execution**
- **portable across OS, independent to OS**
- no executable file produced
- stops at first error
- Python, JS, Ruby

**Compiler**: translates entire program into machine code before execution:
- slow compilation, **fast execution**
- executable is **platform-specific**, must recompile on different OS
- produces a standalone executable
- reports all errors at once after scan
- C, C++, Rust

**Bytecode Interpreter/VM** - source compiled into **intermediate** **bytecode** (.class), then executed by a virtual machine (JVM, Python VM, .NET CLR). Balances between the two above in terms of error detection, translation time, portability and applicability.

**JIT compilation** - a hybrid: compiled bytecode initially interpreted then runtime detects "hot" (frequently executed) code and compiles it to native machine code on the fly. 

**Example scenarios where the translation method should be considered must include rapid development and testing, performance-critical applications and cross-platform development.**

> [!NOTE] Exam Tip
> This is perfect for an "evaluate question"
- Rapid development / testing / prototyping, needs cross-platform -> interpreter, gives immediate feedback
- Performance-critical -> compiler, really need speed and minimal latency
	- this is why C++ is super fast and python is not
- Middleground -> bytecode + VM + JIT
